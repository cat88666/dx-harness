# 主题-德州蘑菇逃庄留座离桌链路

## 当前结论

当前事实来自 `dx-game-texas` 本地分支 `duik-mushroom-temp-final`。本分支已经不是“待设计拦截点”状态，而是已引入 `MushroomNextButtonService` 作为统一的下一手蘑菇庄位保护服务。

需求语义：蘑菇玩法进行中，玩家如果是下一手庄位且筹码充足，则在“本手结算到下一手开始”窗口内不能离桌。留座离桌 BUG 的关键修复点已经覆盖到结算后执行路径：命中保护时取消 `nextLeave`、发送 `TEXAS_MUSHROOM_NEXT_BUTTON_CANNOT_LEAVE` 提示，并且不迁移到占座列表。

## 保护窗口

窗口由两个源码点共同定义：

```text
HandOverListener.handleSettlement(...)
  -> MushroomNextButtonService.handNextButton(tableHand)
  -> tableHand.nextButtonUserId = 下一手庄位玩家

GameStartHandlerBase.doRoomGameStart(...)
  -> tableHand.setNextButtonUserId(null)
```

也就是说，当前实现是在本手结算阶段写入 `nextButtonUserId`，下一手开始时清空。早期设计文档中提到的 `TableCommonService.initNextHandData` 不是当前实际调用点，后续判断应以源码为准。

## 判断条件

统一判断入口是：

```text
MushroomNextButtonService.isMushroomNextButtonBlocked(tableId, userId)
```

命中保护需要同时满足：

- `TableInfoService.isStackMushroomEnabled(tableInfo)`：堆蘑菇开关开启。
- `TableMushroom` 存在且 `tableMushroom.isStart()`：蘑菇轮次进行中。
- `tableHand.nextButtonUserId == userId`：玩家是下一手庄位。
- 玩家仍在 `TableUserCache.getTableUsers(tableId)` 中：避免幽灵引用。
- 当前参与玩家数量大于 1：单人桌不保护。
- 玩家筹码满足 `chipScore >= TableNextChipService.getNextButtonMushroomChipScore(...)`。

封顶场景需要注意：`TableNextChipService.getNextButtonMushroomChipScore` 在蘑菇池封顶时返回 `BigDecimal.ZERO`，因此只要玩家筹码非负就会视为筹码充足，仍会拦截。这与回归测试点中“蘑菇池封顶仍需保护”的方向一致。

## 留座离桌链路

用户点击留座离桌入口：

```text
TexasProto.USER_HOLD_SEAT
  -> UserHoldSeatCommand.handleCommand
  -> UserHoldSeatCommand.handle
```

当前行为分两类：

- 若玩家不在手牌开始或结算阶段，或已经弃牌，则直接调用 `UserHoldSeatHandler.immediatelyHoldSeatLeave(..., false)`。
- 若玩家处于 `GAME_HAND_START` 或 `GAME_SETTLEMENT`，则设置 `userItem.nextLeave = true`，等待结算后执行。

结算后真正执行留座离桌的位置：

```text
UserHoldSeatHandler.immediatelyHoldSeatLeave(table, userItem, true)
```

该路径当前已经接入蘑菇保护：

- 先执行 calltime 保护。
- 再调用 `MushroomNextButtonService.isMushroomNextButtonBlocked(tableId, userId)`。
- 命中后 `userItem.setNextLeave(false)`。
- 调用 `MushroomNextButtonService.sendMushroomNextButtonToast(tableId, userId)`。
- 直接 `return`，不设置 `systemHold = USER_LEAVE`，不发布 `UserLeaveTask`，不把玩家从 `tableUsers` 移到 `occupyUsers`。

次数消耗风险已和旧判断不同：`UserHoldSeatCommand.handleCommand` 虽先执行 `userHoldSeatHandler.tryAcquire(...)`，但真正的 `UserRateLimiterCache.putLimiter(... HOLD_SEAT)` 在 `UserHoldSeatHandler.immediatelyHoldSeatLeave` 成功迁移到占座后执行。结算后蘑菇拦截在迁移前返回，不会消耗一次留座离桌次数。

## 已接入出口

`MushroomNextButtonService` 不是只服务留座离桌，当前分支已经把多个离桌出口收敛到同一判断。

| 出口 | 源码落点 | 当前行为 |
| --- | --- | --- |
| 结算后留座离桌 | `UserHoldSeatHandler.immediatelyHoldSeatLeave(..., true)` | 命中后清 `nextLeave`、toast、阻止迁移到占座 |
| 非结算后立即留座离桌 | `UserHoldSeatHandler.immediatelyHoldSeatLeave(..., false)` | 通过 `checkMushroomNextButton(HOLD_SEAT_COUNTDOWN, ...)` 抛 toast |
| 留座倒计时到期强制站起 | `UserLeaveTask.run` | 命中后清 `nextLeave`、toast，并执行 `USER_BACK` |
| 普通预站起设置 | `PrepStandCommand.handle` | 先写 `prepStandStatus`；若当前不在手牌进行中，会同步触发 `SpecifyService.handleForcedUserAction` |
| 普通预站起执行 | `SpecifyService.handleForcedUserAction` | `PREPARE_STAND` 分支命中蘑菇保护时取消预站起并 toast |
| 主动站起 / 系统站起 | `UserStandHander.playerStand` | 按 `KickerReasonEnum` 白名单判断 |
| 俱乐部主动退出 | `ClubUserQuitService.playerQuit` | commonCheck 与 calltime 后判断 |
| 私人房主动退出 | `PrivateUserQuitService.playerQuit` | commonCheck 与 calltime 后判断 |
| 切桌 / MTT 切桌退出 | `UserQuitServiceFactory.buildRaceQuitContext` | 构造退出上下文时判断 |
| 鱿鱼预站起结算执行 | `UserStandUpService.handlePreStandBySquid` | 命中后取消鱿鱼预站起并 toast |
| 鱿鱼结算后强制站起 | `SquidSettlementAfterService.handStand` | stand 列表过滤掉被保护玩家 |
| 超时托管 3 手站起 | `TableCommonService.betTimeOutStand` | 命中后 return，不执行站起 |

## 白名单边界

`MushroomNextButtonService.MUSHROOM_BUTTON_GUARD_LIST` 是当前保护边界。白名单覆盖：

- 大厅换桌主动退出 / 站起
- 普通预站起
- 鱿鱼预站起
- 无操作站起
- 留座离桌倒计时
- 主动退出
- 主动站起
- 前往比赛站起 / 退出
- 受邀预站起 / 预退出
- 3 手超时站起

不在白名单的原因会记录日志并放行。当前注释明确不拦截的场景包括房主踢人、牌桌风控、牌桌解散、旁观超限、断连清理、补码。后续新增离桌 reason 时，必须同时审查是否需要加入 `MUSHROOM_BUTTON_GUARD_LIST`，否则会形成新的逃庄出口。

## 仍需验证点

- 普通预站起设置阶段本身不直接调用 `MushroomNextButtonService`，但当前已确认有两条兜底：手牌进行中只写 `prepStandStatus`，结算后由 `SpecifyService.collectPreStandUsers` 收集并在 `handleForcedUserAction` 的 `PREPARE_STAND` 分支拦截；非手牌进行中会在 `PrepStandCommand.handle` 内同步触发 `SpecifyService.handleForcedUserAction`，同样走蘑菇拦截。
- `SquidSettlementAfterService.handStand` 对超时强制站起列表是过滤保护玩家，不一定直接 toast；测试点目前更关注“不被强制站起”。
- `MushroomNextButtonService.handNextButton` 仅在堆蘑菇开关开启时写 `nextButtonUserId`。如果后续支持结算窗口内动态改配置，需要另查配置刷新链路。
- 当前未发现针对该保护服务的 JUnit 单测；验收资料主要是 `docs/蘑菇防逃庄功能回归测试点.md` 中 TC-01 到 TC-22 的人工/回归测试点。

## 后续开发定位

若继续查 BUG 或补回归，优先按这个顺序：

1. `MushroomNextButtonService`：统一判断、toast、白名单、下一手庄位计算。
2. `HandOverListener.handleSettlement`：结算阶段写入 `nextButtonUserId`。
3. `GameStartHandlerBase.doRoomGameStart`：下一手开始清空 `nextButtonUserId`。
4. `UserHoldSeatCommand` 与 `UserHoldSeatHandler`：点击留座离桌和结算后真正迁移。
5. `UserLeaveTask`：留座倒计时到期后的强制站起。
6. `PrepStandCommand` 与 `SpecifyService`：普通预站起设置、收集和执行。
7. `UserStandHander`、`ClubUserQuitService`、`PrivateUserQuitService`、`UserQuitServiceFactory`：其他离桌出口。
