# 主题-德州蘑菇逃庄留座离桌链路

## 链路定位

“结算留座离桌时正好处于蘑菇庄位，不能留座离桌”应定位到 `dx-game-texas`：

```text
USER_HOLD_SEAT
  -> UserHoldSeatCommand.handleCommand
  -> UserHoldSeatCommand.handle
  -> userItem.nextLeave = true
  -> 结算后流程
  -> UserHoldSeatHandler.immediatelyHoldSeatLeave(..., true)
  -> TableUserCache.tableUsers -> occupyUsers
```

蘑菇庄位相关链路：

```text
GameStartHandlerBase.doRoomGameStart
  -> findUserService.selectButtonBbSbStraddle(...)
  -> tableHand.buttonUserId
  -> mushRoomGameService.addMushRoom(tableId, buttonUser)
  -> TableMushroom.curUserMushroom / userMushroomMap
```

## 关键判断

逃庄 BUG 的风险点不在大厅配置展示，而在结算后真正执行留座离桌时，玩家已经被允许从 `tableUsers` 移到 `occupyUsers`。如果该玩家下一手将成为蘑菇庄位，开局时 `addMushRoom` 无法再按庄位扣蘑菇，形成逃庄。

## 候选拦截点

优先拦截点:

- `UserHoldSeatHandler.immediatelyHoldSeatLeave(Table table, UserItem userItem, Boolean isSettleAfter)`

理由:

- 这是结算后真正移动用户状态的位置。
- `isSettleAfter = true` 时已经有 calltime 的结算后取消逻辑，适合补同类“结算后不能离开”的规则。
- 拦截时可以将 `userItem.nextLeave` 置回 `false`，避免后续重复触发。

备选补充点:

- `UserHoldSeatCommand.handle` 可增加主动点击时的早期提示，但它无法覆盖“结算时状态变化后才发现下一手庄位”的场景。

## 判断所需数据

需要从现有代码确认一个稳定的“下一手蘑菇庄位”判断方式：

- `FindUserService.getNextHandSeat(tableId)` 是否能在结算后、开局前稳定返回下一手 `buttonUserId` 或 button chair。
- 或复用 `findUserService.selectButtonBbSbStraddle` 前置计算中的同源逻辑，避免另写一套选庄规则。
- `TableMushroom.isStart()` 可判断蘑菇轮是否已经有池子；若需求要求“蘑菇游戏中”才限制，应结合 `TableInfo.stackMushroomSwitch == 2` 与 `TableMushroom` 状态判断。

## 推荐修复形态

新增一个小的业务判断方法，例如放在 `MushRoomGameService` 或独立服务中：

```text
boolean isNextMushroomButton(Long tableId, Long userId)
```

该方法只负责判断，不直接变更用户状态。`UserHoldSeatHandler.immediatelyHoldSeatLeave(..., true)` 调用它，命中后：

- 记录日志，带 `tableId`、`userId`、`handCode`、`next button` 信息
- `userItem.setNextLeave(false)`
- 发送或抛出 `TexasToastCode.TEXAS_MUSHROOM_NEXT_BUTTON_CANNOT_LEAVE`
- 不移动到 `occupyUsers`
- 不发布 `UserLeaveTask`
- 留座离桌次数需要单独确认：当前 `UserHoldSeatCommand.handleCommand` 是先 `tryAcquire` 再 `handle`，如果拦截放在结算后，需要补偿限流次数，或把蘑菇庄位判断前移到消耗次数之前

## 需要补测试的场景

- 蘑菇开关关闭：不拦截。
- 蘑菇开关开启但玩家不是下一手庄位：允许结算后留座离桌。
- 蘑菇开关开启且玩家是下一手庄位：禁止结算后留座离桌，并清理 `nextLeave`。
- 玩家已弃牌、非结算后立即离开路径：确认是否也需要同样限制，避免只修结算路径留下主动逃庄入口。
