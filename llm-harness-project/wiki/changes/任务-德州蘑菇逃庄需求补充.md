# 任务-德州蘑菇逃庄需求补充

## 需求结论

在德州堆蘑菇玩法中，修复“上一次逃庄”的补充 BUG：玩家在本手结算后触发留座离桌时，如果该玩家正好是下一手蘑菇庄位且筹码充足，则不能留座离桌。

该需求属于 `dx-game-texas` 的牌桌内行为，不属于 `dx-game-hall`。

## 当前上下文

- 当前本地分支: `duik-mushroom-temp-final`
- 蘑菇开发分支族: `feture-sun-16603`、`origin/16603-final`、`origin/duik-mushroom-temp-final`
- 用户明确上下文: “当前德州分支正在开发和提测第三轮的蘑菇游戏开发和蘑菇逃庄需求补充”
- 当前实现状态: 已有 `MushroomNextButtonService`，并已接入留座离桌、主动站起、主动退出、切桌/比赛退出、鱿鱼预站起、超时强制站起等出口
- 关键目标: 用该 BUG 打通德州游戏开发 harness 流程，不只是修一个点

## 问题边界

留座离桌原链路中：

- `USER_HOLD_SEAT` 入口由 `UserHoldSeatCommand.handleCommand` 处理
- 玩家在手牌进行中或结算中会设置 `userItem.nextLeave = true`
- 结算后由 `UserHoldSeatHandler.immediatelyHoldSeatLeave(..., true)` 完成真正留座离桌
- 真正迁移成功后，玩家会从 `tableUsers` 移到 `occupyUsers`

逃庄风险就在结算后真正迁移前。如果下一手庄位玩家被移动到占座列表，下一手开局的蘑菇扣取链路就无法按庄位正常执行。

当前分支的修复方式是提前在结算阶段计算 `tableHand.nextButtonUserId`，并在所有白名单离桌出口调用 `MushroomNextButtonService` 判断是否禁止离桌。

3. 查 `MushroomNextButtonService`，确认白名单、筹码判断、toast 和 `nextButtonUserId` 计算逻辑。
4. 查 `HandOverListener.handleSettlement` 与 `GameStartHandlerBase.doRoomGameStart`，确认保护窗口是否正确写入和清空。
5. 若有新逃庄入口，先检查该入口的 `KickerReasonEnum` 是否在 `MUSHROOM_BUTTON_GUARD_LIST` 中，再检查对应服务是否调用 `checkMushroomNextButton`。

## 当前验收要点

- 蘑菇开关未开启时，原离桌行为不变。
- 蘑菇开关开启但玩家不是下一手庄位时，原离桌行为不变。
- 玩家是下一手庄位但筹码不足时，不应被保护。
- 玩家是下一手庄位且筹码充足时，结算后留座离桌不能移动到占座列表。
- 拦截后需要清理 `nextLeave`，避免后续重复触发。
- 提示复用 `TexasToastCode.TEXAS_MUSHROOM_NEXT_BUTTON_CANNOT_LEAVE`。
- 蘑菇池封顶时仍需保护；当前实现通过 `TableNextChipService.getNextButtonMushroomChipScore` 返回 `BigDecimal.ZERO` 达成。
- 测试资料以 `docs/蘑菇防逃庄功能回归测试点.md` 的 TC-01 到 TC-22 为主。

## 需要持续关注

- `MUSHROOM_BUTTON_GUARD_LIST` 是保护边界，后续新增离桌原因必须同步审查。
- `docs/handleNextButton需求分析.md` 是早期设计背景，其中部分调用点与当前源码不同；当前事实以 `HandOverListener.handleSettlement` 写入、`GameStartHandlerBase.doRoomGameStart` 清空为准。
- 普通预站起路径已确认：`PrepStandCommand` 设置 `prepStandStatus`；手牌进行中等待结算后由 `SpecifyService.collectPreStandUsers` 收集并拦截，非手牌进行中则同步触发 `SpecifyService.handleForcedUserAction` 拦截。
