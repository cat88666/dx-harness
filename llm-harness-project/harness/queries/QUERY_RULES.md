# Query Rules

## 目标

让 `query / locate / runbook` 三类操作稳定输出，不退化成临时聊天。

## `/query`

用于回答“某个链路为什么这样设计、某个问题如何判断”。

要求：

1. 优先读取 `summaries/`、`synthesis/`、`runbooks/`
2. 再补充 `entities/` 中的具体对象
3. 输出必须带出至少一个可直达页面或源码对象
4. 如果知识缺失，应说明缺口，并建议补写哪类页面

## `/locate`

用于定位仓库、服务、模块、类、任务、表、配置项或协议入口。

要求：

1. 先命中 `index.md`
2. 再命中 `wiki/entities/`
3. 输出重点是“去哪看”，包括知识页、真实仓库路径、关键源码入口
4. 不做大段解释

## `/harness-locate`

用于最低 token 成本定位“对象应该去哪看”。

要求：

1. 第一跳只能读 `index.md`
2. 命中后最多读取 1-2 个已链接知识页
3. 输出真实仓库路径、当前分支、相关 feature 分支、关键知识页、关键源码入口
4. 如果没有命中，明确说“索引缺失”，并建议先执行 `/ingest-map <source>`
5. 不做全仓 `rg`，除非用户明确要求或索引明显过期

## `/runbook`

用于按症状返回排障步骤。

要求：

1. 直接返回检查顺序
2. 必须包含高概率原因
3. 必须给出关键源码入口或配置入口
4. 若已有事故页，应关联事故页
